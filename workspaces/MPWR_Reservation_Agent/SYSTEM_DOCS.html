⚡ Version 2.0 — Webhook Orchestrator Release

MPWR Reservation Agent

A fully autonomous system that manages the entire reservation lifecycle between TripWorks and MPOWR — creating, cancelling, rescheduling, and settling payments without manual intervention.

Last Updated: April 16, 2026 Maintainer: Epic 4x4 Adventures Engineering Stack: Python · Playwright · Google Sheets · Slack

✨ Capabilities

🆕

Create Reservations

When a customer books on TripWorks, the agent automatically creates the matching reservation in MPOWR by filling out the web form.

🚫

Cancel Reservations

When a booking is cancelled in TripWorks, the agent cancels it in MPOWR using "Customer Reason" with no cancellation fee.

🔄

Reschedule & Update

When dates, vehicles, or activity types change, the agent updates the MPOWR reservation using the "Great Reset" strategy.

💰

Settle Payments

When a TripWorks balance is paid in full, the agent settles it in MPOWR using Cash. Partial payments are intentionally skipped.

👁️

Monitor Rental Status

Every 30 minutes, checks MPOWR for rental status changes (Checked Out → Returned) and updates the Guest Database.

📊

Guest Database Sync

Maintains a V2 Guest Database in Google Sheets with 47+ columns of normalized data for every reservation.

ℹ️

What it does NOT do: Store credit card info in MPOWR (all settlements use Cash) · Charge cancellation fees · Process partial payments · Create reservations for test bookings, TripAdvisor Exclusives, Pro XPeriences, or Guide Car Passengers.

🏗️ Architecture

The system connects three external platforms (TripWorks, MPOWR, Slack) through a local Python daemon running on a Windows machine, using an ngrok tunnel to receive webhooks.

📋 Data Flows

🆕 New Booking (Creation)

1

Customer books on TripWorks

2

TripWorks fires webhook POST → ngrok → webhook_server.py saves to both webhook_cache/ and webhook_sniffer/

3

Zapier detects TripWorks event, appends row to Google Sheet (Zapier Feed)

4

main.py scheduler triggers run_creator_batch() every 5 minutes

5

sheets.py scans Google Sheet for rows without MPWR Confirmation Number

6

data_mapper.py translates row into MPOWR form values (activity, vehicle, dates, pricing)

7

mpowr_creator_bot.py opens Chromium, logs into MPOWR, fills the form, and submits

8

Bot extracts MPOWR Confirmation ID → writes to Secondary Tracker + V2 Guest Database

9

Slack success notification sent with pre-submit screenshot

🚫 Cancellation

1

Staff or customer cancels the booking in TripWorks

2

TripWorks fires webhook with status.slug = "cancelled"

3

webhook_processor.py looks up the MPWR Confirmation Number in V2 Guest Database

4

Bot navigates to MPOWR order page → clicks Actions → Cancel Reservation

5

Selects "Customer Reason" and ensures "Charge Cancellation Fee" is UNCHECKED

6

Clicks Cancel Order → updates V2 Dashboard (Status = "Canceled") → sends Slack alert

🔄 Reschedule / Update

1

Staff or customer changes dates, vehicles, or activity type in TripWorks

2

TripWorks fires webhook with status.slug = "booked" (same as creation!)

3

webhook_processor.py finds an existing MPWR ID → classifies as UPDATE, not creation

4

Bot navigates to order → clicks Actions → Reschedule

5

Sets new activity listing, dates, and time

6

"The Great Reset" — zeros ALL vehicle dropdowns to prevent double-booking

7

Applies new vehicle type, quantity, and overrides price → clicks Update Reservation

💰 Payment Settlement

1

Customer pays remaining balance on TripWorks

2

webhook_processor.py checks if trip.due == 0 (paid in full)

3

If partial → SKIP (Slack notifies team). If paid in full → proceed

4

Bot clicks Actions → New Charge → Make Payment → Cash → Collect

5

Updates V2 Dashboard: Amount Due = $0.00, Status = "Paid" → sends Slack alert

⚠️

Important: We NEVER enter customer card information into MPOWR. All settlements are processed through the "Cash" option only. Partial payments are always skipped.

📦 Modules

🎯 main.py — Daemon Orchestrator

The entry point for the entire system. Uses APScheduler with Denver timezone to run jobs on cron schedules.

JobScheduleFunction
Creator (Peak)Every 5 min, 7AM–6:59PMrun_creator_safely()
Creator (Off-Peak)Every 60 min, 7PM–6:59AMrun_creator_safely()
Ride CheckerEvery 30 minrun_ride_checker()

Each run_creator_safely() cycle first sweeps webhooks for cancel/pay/update events, then scans Google Sheets for new bookings.

📡 webhook_server.py — HTTP Listener

Lightweight HTTP server on port 8000, exposed via ngrok. On every POST:

  1. Parses JSON payload
  2. Saves to webhook_cache/{conf_code}.json (legacy compatibility)
  3. Saves to webhook_sniffer/{timestamp}_{conf_code}.json with full HTTP headers
  4. Returns HTTP 200

🧠 webhook_processor.py — Event Router

The brain of the webhook system. Classifies every incoming event and routes it to the correct bot action.

For each .json file in webhook_sniffer/:
  1. Hash payload (MD5) → skip if already processed
  2. Extract TW Confirmation Code
  3. Early-classify: Payment? Cancellation?
  4. Look up MPWR ID in V2 Guest Database
  5. No MPWR ID + Payment/Cancel → pending_retry/
  6. Has MPWR ID → route to settle/cancel/update
  7. Record hash → move to processed/

🤖 mpowr_creator_bot.py — Playwright Engine

The largest module (~2050 lines). Controls a headless Chromium browser.

MethodPurpose
create_batch(payloads)Creates multiple reservations sequentially
create_reservation(payload)Fills and submits the /orders/create form
cancel_reservation(mpwr_id)Actions → Cancel → Customer Reason → $0 fee
settle_payment(mpwr_id)Actions → New Charge → Cash → Collect
update_reservation(mpwr_id, payload)Actions → Reschedule → Great Reset → Update
🔄

"The Great Reset": When editing a reservation, the bot sets ALL vehicle quantity dropdowns to 0 before applying new selections. This prevents double-booking when a customer switches vehicle types. Includes a keyboard-input fallback if MPOWR ever replaces native HTML selects.

🗺️ data_mapper.py — Translation Layer

875 lines of mapping tables that translate between TripWorks terminology and MPOWR form values. Handles activities, vehicles, ticket types, pricing, time slots, guide add-ons, and insurance.

Intentional Skip Rules

The bot skips (and Slack-notifies) these reservation types:

  • Test reservations (name contains "test")
  • TripAdvisor Exclusive activities (managed natively)
  • Pro XPerience activities (no MPOWR inventory needed)
  • Guide Car Passengers/Riders only (no vehicle allocation)

📊 sheets.py — Google Sheets Integration

Manages three separate Google Sheets:

SheetPurpose
Zapier FeedPrimary data source (read-only by bot)
Secondary TrackerTracks which TW Confirmations have been processed
V2 Guest Database47-column normalized database (the operational dashboard)

💬 slack_notifier.py — Team Notifications

Sends rich Block Kit messages to Slack via Bot Token (preferred) or Incoming Webhook (fallback). Falls back to console if neither is configured. Supports screenshot uploads via Slack's modern 3-step file upload API.

🛡️ Safety Mechanisms

🔒 Webhook Deduplication

TripWorks sends 5-10 identical webhooks per event. MD5 hash comparison prevents duplicate processing.

⏳ Race Condition Protection

Cancel/Payment webhooks that arrive before MPWR ID exists go to pending_retry/ and are re-evaluated on the next cycle.

🔄 The Great Reset

During updates, ALL vehicle dropdowns are zeroed before applying new selections, preventing accidental double-bookings.

💲 Fee Guard

The cancellation fee checkbox is explicitly verified as UNCHECKED. If already unchecked (default), no action is taken.

📊 Batch Cap (20/cycle)

Maximum 20 reservations per 5-minute cycle prevents session timeouts and runaway API consumption.

📅 Past Date Filter

Reservations with past Activity Dates are auto-skipped — no ghost bookings get created.

🛑 Circuit Breaker

After 3 consecutive failures, the bot aborts the remaining batch queue to prevent cascading errors.

📸 Screenshot Forensics

Every error captures a full-page screenshot. Auto-cleaned after 7 days.

🚀 Startup & Deployment

First-Time Setup

  1. Clone repository to local Windows machine
  2. Create .env file with all required variables (see Configuration section)
  3. Place Google OAuth client_secret.json in the project root
  4. Double-click Run_Live_Bot.bat — it auto-creates the venv, installs deps & Chromium

Daily Operation

Double-click Run_Live_Bot.bat. The system automatically:

  1. Activates the Python virtual environment
  2. Starts the ngrok tunnel (persistent domain)
  3. Starts the webhook server on port 8000
  4. Starts the main daemon with APScheduler
  5. Runs an initial boot sweep immediately, then operates on cron schedule
✅

To stop: Close the terminal window or press Ctrl+C.

⚠️ Known Limitations

1️⃣

MPOWR UI Sensitivity: All Playwright automation relies on specific button text and CSS selectors. If Polaris updates the MPOWR interface, selectors may need updating.

2️⃣

Single Machine: The system runs on one local Windows machine. If the machine is off, webhooks are lost after the ngrok tunnel disconnects.

3️⃣

Undocumented API: All TripWorks webhook event classification is reverse-engineered. Payload structures may change without notice.

4️⃣

Hash file growth: processed_hashes.txt grows indefinitely. Consider periodic cleanup of entries older than 30 days.

5️⃣

No multi-vehicle update: The update path assumes a single vehicle type per reservation. Complex multi-model swaps are not yet supported.
